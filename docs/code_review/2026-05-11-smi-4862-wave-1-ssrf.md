## Findings

### 1. CRITICAL (fixed): `..;/` path-confusion bypasses all validation checks

**File**: `apps/api-proxy/api/proxy.ts`, line 53 (original)

The input `functions/v1/..;/etc/passwd` passes every validation step in the original commit:
- No leading `/`
- No `:`
- No literal control bytes
- `new URL('functions/v1/..;/etc/passwd', base)` resolves to pathname `/functions/v1/..;/etc/passwd` — the URL spec treats `..;` as a non-dot-segment (extra characters past the two dots), so it is NOT normalised away
- Origin is unchanged (stays on Supabase)
- `startsWith('/functions/v1/')` is true

Whether this is exploitable depends on whether Supabase's upstream (Deno edge runtime, load balancer, or nginx) normalises `..;/` as a traversal. nginx in `merge_slashes off` mode does. The OWASP Web Security Testing Guide lists this as a required rejection. Fix: pre-parse `/(^|\/)\.\.([/;]|$)/` check.

### 2. MAJOR (fixed): Percent-encoded control characters bypass the literal-byte regex

**File**: `apps/api-proxy/api/proxy.ts`, line 37 (original)

The control-char check `/[\x00-\x1f]/` matches literal 0x00–0x1f bytes. The input `functions/v1/foo%00bar` contains `%`, `0`, `0` — not a literal NUL — so it passes. After URL construction the pathname contains `%00` (preserved as-is), which Deno decodes to NUL when routing, almost certainly resulting in a 404. Similarly for `%0d` (CR) and `%0a` (LF), which could be smuggled into headers by a non-hardened intermediary. Fix: added `/%0[0dDaA]/i` rejection.

### 3. MAJOR (fixed): Dead-code `resolved.pathname.includes('/../')` post-parse check

**File**: `apps/api-proxy/api/proxy.ts`, line 53 (original)

The URL constructor resolves dot-segments per RFC 3986. After `new URL('functions/v1/../etc/passwd', base)` the pathname is `/functions/etc/passwd` — no `/../` substring exists. This check can never be true. It gave false confidence while the real traversal defense was the `pathOk` prefix check. Removed; replaced with an explanatory comment documenting why post-parse dot-segment checks are unnecessary here (pre-parse rejection + prefix check are sufficient).

### 4. MINOR (fixed): Test suite missing attack vectors for the above bypasses

**File**: `tests/api-proxy/proxy.test.ts`

The `it.each` attack-vector table listed 8 vectors but did not include `..;/`, `%00`, `%0d`, or `%0a`. Added 4 cases with inline comments explaining the bypass mechanism each covers. Suite is now 19/19.

## Items Confirmed Clean (no action required)

**URL origin check**: `resolved.origin !== baseOrigin` uses string equality on canonical origins (scheme + host + port). IPv6 brackets and unicode hostnames are normalised by the URL constructor before comparison. No bypass found.

**`@` in path position**: `functions/v1/foo@evil.com/bar` — the `@` appears after a path segment, not before the host, so the URL constructor parses it as a path character, not userinfo. Origin is unchanged. Test already covers this.

**Percent-encoded traversal `%2f..%2f`**: `new URL('functions/v1/foo%2F..%2Fbar', base)` resolves to `/functions/v1/foo%2F..%2Fbar` — `%2F` is not decoded by the URL constructor path normalization, so `..` is not treated as a segment. Stays within `/functions/v1/`. Upstream decides what `%2F` means; we stay on-origin. Test covers this.

**Header forwarding**: The forwarded set (`authorization`, `apikey`, `content-type`, `x-request-id`) is allow-listed, not forwarded wholesale. Node.js's HTTP parser rejects CRLF in header values at the wire level before they reach the handler. No injection risk in the header forwarding logic.

**`targetUrl` source**: `resolved.toString()` is called on the URL object after all validation passes, not on the original `path` string. Fetch operates on the validated URL, not on user input.

**Allow-list completeness**: `/functions/v1/` and `/rest/v1/` cover the two paths this proxy was built to expose (edge functions and PostgREST). Storage (`/storage/v1/`), Realtime (`/realtime/v1/`), Auth (`/auth/v1/`), and GraphQL (`/graphql/v1/`) are intentionally excluded — exposing them would be a scope expansion requiring a separate decision. The narrow allow-list is the correct security posture here.
